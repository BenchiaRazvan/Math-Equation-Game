1. Introduction

1.1 Purpose

The current version number for this product is 1.0 (first version). The product is intended for learning the basics of mathematics, in the form of interactive exercises.

1.2 Document Conventions

  • The document is based on the template IEEE System Requirements Specification Template .
  • Sequences of references are specified in italics.

1.3 Intended Audience and Reading Suggestions

The document is intended for users, developers and testers. We suggest reading the document for users in the current format, starting with the introduction. For developers and testers, we suggest reading the document with the "Overall Description".

1.4 Product Scope

The aim for the user is to accumulate knowledge about different fields of mathematics (algebra, geometry, trigonometry, analysis), by age categories (primary, middle school, high school). The problems presented in the application are divided into modules - light, medium and a difficult problem, which requires addiotional research on the solution technique.

  • H Rick. IEEE-Template. GitHub
  • Scholarly HTML w3c.gihtub.io

2. Overall Description

2.1 Product Perspective

This product is presented as a new product, autonomous, but still similar to sites for others fields (computer science, biology etc.). It shares the same ideas with many other sites, but for the fields of mathematics.

2.2 Product Functions

  • Math problem solving
  • Registration in the application
  • Application authentication
  • Sending feedback about application
  • Editing the user profile

2.3 Uses Classes and Characteristics

We believe that the class of users who will use this application will be mainly composed of students from primary school, middle school and high school. They can solve interactive problems, being rewarded for each problem solved correctly with virtual coins.

2.4 Operating Environment

For optimal use of the application, version 1.0, we recommend using the Mozilla Firefox browser (other browsers may not display properly). Also, there is no need for top hardware specifications - due to the simplicity of the application, we believe that it can run on the vast majority of computers.

2.5 Design and Implementation Constraints

For the web application interface, Math Equation Game, we used only HTML5, CSS3, JavaScript and Ajax. Instead, to create the backend we will mainly use PHP. The developer is constrained by the fact that presenting problems to the client requires the use of the MathML utility. This utility contains certain "components" that do not display to customer information correctly, so little tricks have been used.

2.6 User Documentation

In order for the web application to work in optimal parameters, it is necessary to use the Firefox browser. The MathML format is not supported by some browsers, and for this reason, it is recommended to use the Firefox browser.

2.7 Assumptions and Dependencies

The part of the database in which users' information is stored may affect the utility of the application (for example, if the database service requires a fee, depending on the complexity, changes will be made to the application).

3. External Interface Requirements

3.1 User Interface

The components for which a graphical user interface was required are:

  • Home window
  • Login window
  • Register window
  • Contact window
  • Problem list window (separate window for each student category, respectively each category of mathematics field)
  • The admin page, separate from the main site, to which the user does not have access. Only the site administrator can get here, who can add and delete problems.

3.1.1 Home Interface

The image below is the main window, in which the user will enter to authenticate or register. The user has at his disposal a navigation bar, through which he can access the contact page, he can access the proposed problem categories or he can search for information in the application. Also, by using the "login" button, the user will be sent to the page intended for authentication, and by using the "register" button, the user will be sent to the page intended for registration in the application.

3.1.2 Login Interface

The image below represents the authentication window, in which the user can authenticate in the application. The user must enter their username and password to successfully complete the process. The user also has access to the key button, displayed next to the message "You don't have account?", which is redirected him to the registration window.

3.1.3 Register Interface

The image below represents the registration window, where the user has the possibility to create an account. To create an account, the user must completely and correctly fill in all the fields for this service. The service ends successfully when the user presses the "register" button. The user also has the option to set the account profile picture. If the user realizes that he already has an account created in the application, he can press the key button, displayed next to the message "You already have an account?", which will redirect him to the login page.

3.1.4 Contact Interface

The image below represents the contact page, where the user can leave a feedback, an observation or a possible improvement for the application, in the form of a message. The message will reach the application administrators, who will evaluate the received form. The user is obliged to correctly and completely fill in all the fields for this service. To successfully complete the service, the user will press the "submit" button.

3.1.5 Problem list Interface (for primary school)

The image below is the problem window for the primary category, in which the interaction with the problems is done in the form of a game. We considered this method in this category because the primary category represents the youngest users of the application, and a form of play could stimulate them in solving exercises. The user also has access to the "Home" button, which redirects him to the main window, the "Contact" button, which allows the user to leave feedback, and the "Problems" button, which allows the user to access the other categories of problems.

3.1.6 Problem list Interface (for other categories)

The image below is the pattern for the list of problems for middle school and high school. Each category described above also includes different categories of problems (algebra, analysis, geometry, trigonometry). Each area of mathematics comprises five problems - 2 easy, 2 medium and a difficult problem, which require additional research to understand the solving technique. The user has the choice of the field of mathematics in which he wants to solve the proposed problems. Also, to solve a problem, the user must enter the answer in the field intended for him, and after pressing the button next to the field.

3.2 Hardware Interfaces

Devices that can interact with the application include both computers (PCs, laptops, other similar types) and tablets and mobile devices. Because the application has a responsive interface, it can scale information to different screen sizes.

3.3 Software Interfaces

A first software component that establishes a connection with the application is the database. To access the database, the PHP language is used, and to modify the data inside the database, MySQL is used. The application also uses the MathType utility suite, which allows the correct formatting and the correct display of problems in the form of MathML. The application administrator enters the desired equation in the MathType utility, and through certain commands, the utility returns the equation in MathML form, which will be entered in the HTML file, for a correct display of the equation for the user. Also, scripts written in JavaScript language are used to improve the user experience in the application.

3.4 Communications Interfaces

The application imposes the condition that the user has an internet connection, in order to be able to verify the answers given by him. Also, the internet connection is necessary regarding the contact with the application developers, in the contact window. Another protocol that is used within the application (on the server side) is the HTTP protocol.

4. System Features

4.1 System Feature 1: Solving Math Problems

This feature has a high priority because it is the main feature of the application. In addition to the other features, the main purpose of the application is to solve, in an interactive way, math problems in different categories. Problems are considered solved if the user enters the correct answer of the problem in the field intended for the answer.

4.2 System Feature 2: Participation in the Top

This feature has a medium priority, because the top helps users to be recognized in the application, but they have no reward (both in the application and outside it). Ranking in a place as high as possible in the ranking highlights the user as being diligent and determined to understand the techniques for solving the problems described in the application.

5. Other Nonfunctional Requirements

5.1 Performance Requirements

Currently (version 1.0 of the application), the developers consider that there are no performance requirements, as long as the application runs in a browser specified in the previous points (Mozilla Firefox).

5.2 Safety Requirements

The results of testing the application in the current version (1.0) showed that there are no losses, damages or losses that may occur from the use of the application. However, if a violation of one of the above is observed, it is recommended to isolate and analyze the incident.

5.3 Security Requirements

The database used is MySQL, and each user's data is stored in the database. Each user's password is not stored properly, but is applied to a hash, after which it is concatenated with another random string. The final result is stored in the database. Through certain functions that cannot be revealed publicly, each user logs in to his password.

5.4 Software Quality Attributes

The application has as secondary purpose reliability, but also portability on different devices (computer, tablet, phone). This work is done with the help of the responsive interface, but also with the periodic care of the application by the developers. If a significant improvement is needed then a new patch appears, in the form of a version.

Other Requirements

1. Formalization of input data

In order for users' data to be stored correctly in the database, they must have the following formalities:

  • Email termination must be required "@yahoo.com"
  • The answer to a problem must be an integer in order to allow the application to validate the answer. The problems are designed in such a way that the answer is an integer. Example answers: "1", "-1", "0.0".
